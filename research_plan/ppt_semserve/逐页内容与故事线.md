## 全局故事线（一句话串起来）

> 显存竞争是 LLM serving 的常态 → vLLM 只会**整请求抢占**，又粗又盲又是断崖式代价 → 但我们知道 KV 的语义价值高度不均（前序工作已证）→ 这正好能用**操作系统内存管理**讲清楚：PagedAttention 补了虚拟内存，但优先级/cgroup/页面回收/压缩交换还空着 → SemServe 补全这套子系统，用**语义价值**替代 recency 做回收 → 三层架构 + 跨请求 water-filling → 我们已经验证了它的算法前提（block 粒度质量不崩）→ 这是想清楚、且动手验证过的系统提案。

## Slide 2 — 背景：显存竞争是常态，不是边角案例 ⭐

- **版面**：左文字要点；右一张示意图（多个请求的 KV 条形叠加逼近一条「显存上限」红线）。
- **页面要点**：
  - KV cache 大小 ∝ 并发数 × 上下文长度
  - 长文 RAG / 多轮 agent / 8k–32k prompt → 单请求 KV 动辄数 GB
  - 显存是**硬约束**：running 请求 KV 总量逼近上限 → 调度器必须回收 → **抢占（preemption）**
- **故事线**：「大模型推理服务里，多个用户共享一块 GPU 的显存。每个请求的 KV cache 随上下文长度线性增长——一个 8k 到 32k 的长文档请求，KV 就是几个 GB。并发一上来，所有请求的 KV 加起来很快撞到显存天花板。显存是硬约束，撞到了调度器就必须回收显存，这个动作就叫抢占。这不是极端情况，是高负载下的常态。」
- **过渡**：「问题是——vLLM 的抢占方式，有三个毛病。」

## Slide 3 — 痛点：vanilla vLLM 的「整请求抢占」病理 ⭐

- **版面**：三个并排卡片（粒度/信号/代价）；底部留一块给 bonus 图（若服务器跑出抢占病理曲线就贴这里，否则放一句机制说明）。
- **页面要点**：
  - ① **粒度太粗**：整请求级——一个 6000-token 请求被抢，整段 KV swap 或丢弃后**重算 prefill**
  - ② **信号错位**：抢占顺序按 FCFS/LIFO/到达序，**与请求内 KV 的信息价值无关**
  - ③ **代价二元**：recompute 的 TTFT 惩罚是断崖式——亚秒级 → 秒级，p99 尾延迟爆炸
  - *（占位）抢占病理曲线：arrival rate ↑ → 抢占数 ↑、TTFT p99 ↑*
- **故事线**：「vLLM 在显存压力下只能做一件事：挑一个请求，把它整个 KV 丢掉或者换出去，等会儿重新算。三个问题：第一，粒度太粗，是整请求级，没有中间档；第二，挑谁的依据是先来后到，完全不看这个请求里的信息重不重要，可能恰好把最该留的踢掉；第三，代价是二元的——一旦重算，被抢请求的首 token 延迟从几百毫秒跳到几秒，p99 尾延迟直接爆炸。」
- **过渡**：「但『整请求』这个粒度，其实漏掉了一个关键事实。」
- **成熟度**：`[设计中/待实测]`——口头说明：「这页的病理曲线我们正在 Blackwell 服务器上实测；机制层面 PagedAttention 论文和 vLLM 抢占行为已经能支撑这个动机。」

## Slide 4 — 关键观察：KV 的语义价值高度不均 ⭐

- **版面**：左一句结论；右一张「25% 预算质量几乎不掉」的前序工作小图或一个大数字。
- **页面要点**：
  - 前序工作（SemantiCache / OP-SieveKV，HF testbed `[已验证]`）：
  - **25% token 预算下，用语义信号选择性保留 KV，NIAH 质量几乎不掉**
  - 结论：**KV 内部存在大量低价值冗余——而 vLLM 抢占完全没利用这一点**
- **故事线**：「一个请求的 KV 里，system prompt、关键事实、用户的真正问题，和大段填充、寒暄、重复，语义价值天差地别。我们前面的工作已经在 HF 上验证过：只保留 25% 的 token，用语义信号挑哪些该留，大海捞针任务的准确率几乎不掉。这说明 KV 里全是冗余。vLLM 整请求抢占，把这个冗余结构完全浪费了。这就是机会。」
- **过渡**：「那怎么系统性地利用它？我发现这件事可以用操作系统讲得特别清楚——这是今天最关键的一页。」
- **成熟度**：`[已验证]`（前序工作）

## Slide 5 — ⭐核心洞见 + OS 类比表（全场最强一页）

- **版面**：顶部一句 framing；主体是 OS 类比表（左 OS 概念 / 中 SemServe 对应物 / 右状态）；高亮「核心贡献」三行。
- **页面要点**：
  - **framing**：PagedAttention = 把虚拟内存分页搬进了 KV cache；prefix caching = COW 共享页。**但内存子系统的其余部分还空着。**

  | OS 概念 | SemServe 对应物 | 状态 |
  |---|---|---|
  | 虚拟内存 / 分页 | PagedAttention | vLLM 已有 |
  | COW 共享页 | prefix caching | vLLM 已有 |
  | 进程优先级（含 aging） | 语义价值+SLO+aging → 租户优先级 | **核心贡献** |
  | cgroup 内存配额 | 优先级 → 租户级 KV budget | **核心贡献** |
  | 页面回收（LRU） | 按 block 语义分数回收 | **核心贡献** |
  | zswap/zram 压缩内存 | int8 warm tier（demote 不丢弃） | 设计中 |
  | swap to disk | demote 到 CPU | 设计中 |
  | OOM killer | 整请求抢占（最后手段 fallback） | vLLM 原生 |

  - **一句话**：SemServe 补全这个子系统，并用**语义价值**替代**访问时近性（recency）**作为回收信号。
- **故事线**：「PagedAttention 这篇工作做的事，本质是把操作系统的虚拟内存分页机制搬进了 KV cache；prefix caching 就是写时复制的共享页。但是操作系统内存子系统不止分页——还有进程优先级、cgroup 配额、页面回收、压缩交换、OOM killer。这些机制在 KV cache 世界里，目前都还没有对应物。SemServe 就是补全这套子系统。而且关键的一点改造：操作系统页面回收看的是『最近有没有被访问』，也就是 recency；我们换成『语义价值』。这张表的中间三行——优先级、配额、回收——就是这个项目的核心贡献。」
- **过渡（防守点先打预防针）**：「我得强调：类比是 framing，不是 contribution。这张表每一行后面都有真实机制和实验。它甚至直接逼出了两条硬性设计约束，待会儿讲算法时会看到——这恰恰说明类比不是空话。」

## Slide 6 — 方法蓝图：三层架构 ⭐

- **版面**：竖向三层框图（数据自上而下流动），层间标注 IR。
- **页面要点**：
  ```
  请求/租户流入
       │
  ① Profiler 层（admission 时异步, CPU）
     prompt → segment 语义价值 → block 价值      复用 semanticache 信号 + OP MLP
       │  RequestSemanticProfile（语义画像 IR）
  ② Priority + Allocator 层  ★核心贡献
     租户层（慢时钟）：优先级 → cgroup budget       语义+SLO+aging
     块回收层（快时钟）：边际语义价值 water-filling   → demote/evict/preempt
       │  ReclaimPlan（回收计划 IR）
  ③ Runtime 层（vLLM fork 内）
     KVCacheManager：block-table 压缩式部分淘汰
     tier_manager：CPU int8 warm tier 降级
     scheduler：抢占路径接入 allocator
  ```
- **故事线**：「整体三层。第一层 Profiler，在请求排队、还没上 GPU 的时候，在 CPU 上异步算它的语义画像——把 prompt 切成段，每段给一个 0 到 1 的保留价值，system prompt 和用户最新问题强制 pin 住。这层复用我们已有的语义信号代码。第二层是核心，分两级：租户层按优先级分 cgroup 式预算，是慢时钟；块回收层在显存真正不够的那一刻，跨请求做细粒度回收，是快时钟。它输出一个回收计划：收谁的哪些 block、降级谁、实在不行才抢占谁。第三层是真正改 vLLM 的地方，把回收计划落到 KV cache 管理、warm tier 降级和调度器上。两层之间用两个 IR 解耦——这让同一套语义逻辑能同时驱动离线 testbed 和在线 runtime。」
- **过渡**：「核心的算法问题是：显存不够时，到底该收谁的 block？」
- **成熟度**：`[设计中]`（算法层本地可开发；Runtime 层依赖 Phase 0 勘察）

## Slide 7 — 核心算法：跨请求 water-filling ⭐

- **版面**：左公式 + 「凹函数→贪心最优」一张小图（一条递减的边际价值曲线）；右两条硬约束。
- **页面要点**：
  - 每个请求 r 有**价值-预算曲线** V_r(b) = block 分数降序前缀和（pinned 永远排最前）
  - 全局 block 不足时求解：
    ```
    max Σ_r V_r(b_r)   s.t.  Σ b_r ≤ B,  b_r ≥ pinned_r,  b_r ≥ floor_r
    ```
  - **V_r 凹 ⇒ 贪心 water-filling 即最优**：每次从边际价值最低的请求收一个 block（堆上 lazy 贪心，近线性）
  - **两条由 OS 类比直接导出的硬约束**：
    1. 优先级**必须带 aging** → 否则低密度租户被饿死（OS 调度经典 starvation 教训）
    2. **tenant 是预算主体，request 是回收执行单元**（两级：cgroup 配额 + 页面回收）
- **故事线**：「把每个请求的 block 按语义分数从高到低排，做前缀和，得到一条『保留 b 个 block 能拿到多少累计语义价值』的曲线。pinned 的 block——system prompt、用户问题——永远排最前，不会被收。因为是降序前缀和，这条曲线是凹的，边际价值单调下降。凹函数下，经典的注水算法就是最优解：每次从当前边际价值最低的请求收一个 block，用堆做，接近线性时间。这里有两条约束是 OS 类比硬逼出来的：第一，优先级必须带 aging，否则低语义密度的租户会被永久饿死——这正是操作系统调度里 starvation 的经典教训，把它写进设计反而说明我们认真对待这个类比；第二，预算的主体是租户不是请求,租户优先级定 cgroup 层的配额,请求层只负责执行回收。」
- **过渡**：「有人会问：这跟已经有的那些语义 KV 工作有什么区别？」
- **成熟度**：`[设计中]`（纯算法，本地可开发+单测；这是 Phase 3 主体）

## Slide 8 — 差异化定位：五维同时成立的唯一系统 ⭐（可压缩）

- **版面**：对比表（相关工作 / 它占的坑 / 与我们的边界）。
- **页面要点**：

  | 已有工作 | 占的坑 | 与 SemServe 的边界 |
  |---|---|---|
  | KVShare | 跨请求语义**复用/去重** | 我们做**分配**不做复用（复用是可选扩展） |
  | Semantic Scheduling | **请求级**语义排序 | 我们做 **token/block 级**跨请求分配 |
  | SAECache | **prefix cache** 语义淘汰 | 我们动**显存竞争下的活跃请求 KV** |
  | RoleKV | **角色级**淘汰 | 我们是连续语义分数 + 全局分配 |
  | Continuum/CacheTTL | **TTL** 保留 | 我们用语义价值替代 recency |
  | SnapKV/StreamingLLM/H2O | **单请求**无差别压缩 | 我们是**跨请求**差异化分配 |

  - **一句话差异化**：唯一在「显存竞争 + 多租户 + token/block 级 + 语义价值驱动 + 渐进压缩替代整请求抢占」五维同时成立的系统。
- **故事线**：「这个方向已经有不少工作，但每个占的坑都不一样。KVShare 做的是跨请求复用，是去重，我们做的是分配，不是复用。Semantic Scheduling 是请求级的排序，粒度比我们粗。SAECache 动的是 prefix 缓存，不是显存竞争下的活跃请求。SnapKV 这类是单请求无差别压缩，正好是我们的一个 baseline。把这五个维度——显存竞争、多租户、block 级、语义驱动、渐进压缩替代抢占——同时摆上,SemServe 是唯一全中的。任何时候发现实现滑向『复用』或『请求级排序』,我们就停下来重新对齐。」
- **过渡**：「光说设计不够。这个方案的算法前提，我们已经动手验证了。」
- **成熟度**：`[定位/文献]`

## Slide 9 — ⭐保底实证：block 粒度质量 gate（the evidence page）

- **版面**：整页放 `fig1_block_granularity_curve.png`；右下或顶部压一行 gate 判定结论。
- **页面要点**：
  - **问题**：把 token 级语义淘汰粗化到 paged-KV 的 **block 粒度**，质量会不会崩？（这是整个 vLLM 路线的算法前提）
  - **图**：`fig1`——4 条线（full 上界 / token 基线 / block-16 ours / block-32），25% 处 gate 线，**Gate: PASS**
  - **判定**：25% 预算下 **block-16 = token 级的 97.4%**（95.0% / 97.5%），> 95% 阈值 → 路线成立
  - **设置**：Qwen2.5-3B-Instruct fp16，2k 上下文，40 样本 manifest，固定 seed
  - **诚实caveat**：10% 极小预算下 block-16 掉到 32.5%（token 仍 100%）→ 极端预算需 block-8（plan §7 备选）
- **故事线**：「要把这套东西放进 vLLM，有一个绕不过去的前提：vLLM 的 KV cache 是按 block 分页的，一个 block 16 个 token，要么整块留要么整块走。而我们的语义打分是 token 级的。粗化到整块对齐,质量会不会崩？这张图就是答案。横轴是保留预算,纵轴是大海捞针准确率。绿色是我们的 block-16。在 25% 预算这个 gate 点,block-16 达到 token 级基线的 97.4%,过了 95% 的阈值。也就是说,整块对齐的代价极小,vLLM 路线的算法前提成立。要诚实说一点:在 10% 这种极端预算下,块粒度的代价会暴露,block-16 掉到 32.5%——这种情况我们的备选是 block-8。但在 25% 到 50% 的主工作区,块粒度几乎免费。」
- **过渡**：「算法前提验证了。那完整的系统怎么证明它比抢占更好?这是接下来要做的。」

## Slide 10 — 评估蓝图 + quality-aware goodput（the money shot 设计）

- **版面**：上半 money shot 对照实验示意（vanilla 抢占 vs SemServe 渐进压缩，两条 TTFT 曲线）；下半 baseline 清单表 + 新指标。
- **页面要点**：
  - **核心对照（money shot）**：同一抢占触发负载下，vanilla vLLM（整请求抢占） vs SemServe（语义渐进压缩）
  - **预期**：TTFT p99 大幅下降（无重算）、吞吐持平或更高、质量探针正确率下降可控（< 5 pt）；**判定**：TTFT p99 改善 > 30%
  - **Baselines**：vanilla / uniform 无差别压缩 / VTC 公平调度 / oracle 上界
  - ⭐**新指标 quality-aware goodput** = 比例(SLO 达标 ∧ 质量探针答对)——把「快」和「对」统一进一个数
  - 负载：chat（ShareGPT）+ RAG 长文（LongBench 8k–32k）+ NIAH 质量探针；Poisson 到达扫 arrival rate
- **故事线**：「证明它更好的关键实验我叫它 money shot:在同一个会触发抢占的高负载下,对比原版 vLLM 的整请求抢占和我们的语义渐进压缩。预期是 SemServe 的 p99 首 token 延迟大幅下降,因为没有重算,吞吐持平或更高,而被压缩请求的质量损失可控,5 个百分点以内。Baseline 有四条,其中 uniform 是每个请求无差别压一半,用来证明『跨请求差异化分配』确实优于『无差别压缩』;oracle 是用离线标注的完美语义信号,给出上界。我们还提一个新指标,quality-aware goodput:只有既满足延迟 SLO、又答对的请求才算数——把『快』和『对』统一进一个指标,因为压缩换速度但答错了是没意义的。」
