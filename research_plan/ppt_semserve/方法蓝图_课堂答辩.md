# SemServe 方法蓝图 —— 课堂答辩讲解稿

> **一句话主张**：在 LLM serving 显存竞争下，把 vLLM "整请求抢占" 升级为 **token/block 级语义价值驱动的跨请求 KV cache 分配**——低语义密度的 block 先被压缩/降级到 int8 warm tier，高密度请求保持全精度，并用 SLO + aging 做公平性兜底。
>
> **答辩日期**：6/17｜**成熟度标注**：本文用 `[已验证]` / `[设计中]` / `[未来实验]` 标明每部分的状态，答辩时请如实区分。

---

## 0. 三十秒电梯陈述（开场白）

> "现在的大模型推理服务，多个用户共享一块 GPU 的显存。当并发请求的 KV cache 总量超过显存上限，vLLM 只能做一件事：**整个请求抢占**——把某个请求的 KV 全部丢掉，等会儿重新算（recompute），它的首 token 延迟（TTFT）从几百毫秒暴涨到几秒。问题在于，这个决策**完全不看请求里的信息重要不重要**。
>
> 但我们知道：一个请求的 KV cache 里，system prompt、关键事实、用户问题，和大段填充、寒暄、冗余，语义价值天差地别。**SemServe 的想法是：与其整请求抢占，不如跨所有请求，把全局语义价值最低的那些 block 先压缩掉。** 这把二元的"抢/不抢"变成了连续的"语义压缩谱系"。
>
> 而这件事，恰好可以用操作系统的内存管理来讲清楚——这是我今天的主线。"

## 1. 问题与动机（Motivation）

### 1.1 LLM serving 的显存竞争是常态，不是边角案例
- KV cache 随 (并发数 × 上下文长度) 线性增长；长文档 RAG、多轮 agent、8k–32k prompt 让单请求 KV 动辄数 GB。
- 显存是硬约束。当 running 请求的 KV 总量逼近上限，调度器必须回收显存——这就是**抢占（preemption）**。

### 1.2 vanilla vLLM 的抢占病理 `[设计中/待实测]`
vLLM 的抢占有三个问题：
1. **粒度太粗**：整请求级。一个 6000-token prompt 的请求被抢占，要么 swap 整块 KV 到 CPU，要么丢弃后**整段重算 prefill**。
2. **信号错位**：抢占顺序基于 FCFS / LIFO / 到达顺序，**与请求内 KV 的信息价值无关**。可能恰好把最该保留的请求踢掉。
3. **代价是二元的**：recompute 的 TTFT 惩罚是断崖式的——被抢占请求的延迟从亚秒级跳到秒级，p99 尾延迟爆炸。

> **动机实验（要拿到的那张图）**：固定长 prompt，逐步抬高并发 / arrival rate，画出 `抢占次数` 与 `TTFT p99` 随负载的曲线——找到稳定触发抢占的"痛点工况"。这就是"问题真实存在"的证据页。

### 1.3 关键观察：KV 的语义价值高度不均
我们前序工作（SemantiCache / OP-SieveKV）已经在 HF testbed 上验证：在 25% 的 token 预算下，用语义信号（信息密度、角色、注意力代理）选择性保留 KV，NIAH 等任务质量几乎不掉。**这说明 KV 内部存在大量低价值冗余——而 vLLM 的抢占完全没利用这一点。**

## 2. 核心洞见与 OS 类比（全场最强的一页）

**故事主线**：PagedAttention 把操作系统的**虚拟内存分页**搬进了 KV cache；prefix caching 是 **COW 共享页**。但 OS 内存子系统的其余核心机制——**进程优先级、cgroup 配额、页面回收、压缩交换、OOM killer**——在 KV cache 世界还没有对应物。**SemServe 补全这个子系统，并用"语义价值"替代"访问时近性（recency）"作为回收信号。**

| OS 概念 | SemServe 对应物 | 状态 |
|---|---|---|
| 虚拟内存 / 分页 | PagedAttention | vLLM 已有 |
| COW 共享页 | prefix caching | vLLM 已有 |
| 进程创建 + 初始工作集 | 租户 admission 时的初始 KV 分配（"fork"） | 设计中 |
| 动态优先级调度（含 aging） | 语义价值 + SLO 等级 + 等待 aging → 租户优先级 | **核心贡献** |
| cgroup 内存配额 | 优先级 → 租户级 KV budget | **核心贡献** |
| 页面回收（active/inactive） | budget 超限时按 block 语义分数回收 | **核心贡献** |
| zswap / zram 压缩内存 | int8 warm tier（demote 而非丢弃） | 设计中 |
| swap to disk | demote 到 CPU 内存 | 设计中 |
| OOM killer | 整请求抢占（最后手段 fallback） | vLLM 原生 |
| IPC 共享内存 | 跨租户语义复用（KVShare 方向） | 可选扩展，非主贡献 |

**讲解纪律（答辩 Q&A 防守点）**：类比是 framing，不是 contribution。每个映射后面都有真实机制与实验。类比直接导出两条硬约束（见 §4），这恰恰证明类比不是空话。

## 3. 方法蓝图：三层架构

```
                请求/租户流入
                     │
   ┌─────────────────▼──────────────────┐
   │  ① Profiler 层（admission 时异步）    │   复用 semanticache 静态语义信号
   │  prompt → segment 语义价值 → block 价值 │   + OP policy MLP；CPU 上算，与排队重叠
   └─────────────────┬──────────────────┘
                     │  RequestSemanticProfile（语义画像 IR）
   ┌─────────────────▼──────────────────┐
   │  ② Priority + Allocator 层（核心贡献） │
   │  租户层（慢时钟）：动态优先级 → cgroup budget │  语义价值 + SLO + aging
   │  块回收层（快时钟）：边际语义价值 water-filling │  → demote / evict / preempt 指令
   └─────────────────┬──────────────────┘
                     │  ReclaimPlan（回收计划 IR）
   ┌─────────────────▼──────────────────┐
   │  ③ Runtime 层（vLLM fork 内）         │
   │  KVCacheManager：block-table 压缩式部分淘汰 │
   │  tier_manager：CPU int8 warm tier 降级    │
   │  scheduler：抢占路径接入 allocator        │
   └────────────────────────────────────┘
```

### 3.1 Profiler 层 —— 请求语义画像 `[设计中，算法部分本地可开发]`
- 在请求 **admission 时**（排队阶段，与 GPU 计算重叠）异步在 CPU 上算：把 prompt 切成 segment（按角色 + 标点），给每个 segment 一个 `[0,1]` 的保留价值分数；system prompt / 最新 user query 强制 pin。
- 复用现有 `SemanticAnalyzer`（信息密度、角色标签等静态信号）+ 可选的 OP-SieveKV `SegmentRetentionMLP`。
- 关键工程动作：把"打分"从"淘汰执行"中**解耦**出来，使 HF testbed 与 vLLM 路线**共用同一份打分代码**——这样论文里两边的数字才可比。
- 输出 `RequestSemanticProfile{segment_bounds, segment_scores, pinned_bounds, mean_density}`，其中 `mean_density`（请求级聚合语义密度）是 allocator 的核心输入。

### 3.2 Priority + Allocator 层 —— 项目核心贡献 `[设计中，算法部分本地可开发+单测]`
**两级结构**（对应 OS 的 cgroup 配额 + 页面回收）：

- **租户层（cgroup 类比，慢时钟，admission/完成时更新）**：
  ```
  priority(t) = w_sem · mean_density_t      # 语义价值
              + w_slo · slo_class_t          # 租户 SLO 等级（0/1/2，静态配置）
              + w_age · age(t)               # aging：防低密度租户饿死
  budget_t   = clamp(B_total · priority_t / Σ priority,  floor_t,  ceil_t)
  ```
- **块回收层（页面回收类比，快时钟，free block 不足时触发）**：在租户 budget 给出的 `floor_r` 约束下，对所有 running 请求做跨请求的 block 级 water-filling（见 §4）。

**输出**：`ReclaimPlan{reclaim_per_request, unmet_blocks, preempt_candidates}`——告诉 runtime 收谁的哪些 block、还差多少、谁是最后才动用原生抢占的候选。

### 3.3 Runtime 层 —— vLLM fork 内的系统改造 `[设计中，依赖 Phase 0 勘察]`
- **KVCacheManager.compact_request(request_id, keep_block_mask)**：对单个请求做 block-table 压缩式部分淘汰（删中段 block、前移压缩、修正 seq_len / num_computed_tokens）。**这是整个系统的 go/no-go 技术点**（§7 风险表 Task 2.1 spike）。
- **tier_manager（int8 warm tier）**：被回收的 block 先 quantize → demote 到 CPU（复用 `quantized_tier_cache.py`），而非直接丢弃。MVP 先做 **demote-only**（不自动召回），这已构成"渐进压缩 vs 整请求抢占"的完整故事。
- **scheduler 接入**：free block 不够时**先问 allocator**，按 plan 执行 compact/demote；只有 `unmet_blocks > 0` 才退回 vLLM 原生抢占——保证不会活锁（全 pinned 时能 fall through）。

### 3.4 RetentionPlan IR —— 算法与系统的桥梁
`segment 分数 → block 分数 → keep mask` 这条 IR 把"算法决策"和"系统执行"解耦，使同一套语义逻辑能同时驱动 HF 离线 testbed 和 vLLM 在线 runtime。`semserve.block_mapping.segment_scores_to_block_scores` 是这座桥的核心函数（已实现 `[已验证]`）。

## 4. 核心算法：跨请求 water-filling

**抽象**：每个 running 请求 r 有一条**价值-预算曲线** `V_r(b)` = "保留 b 个 block 时的累计语义价值" = block 分数降序后的前缀和（pinned block 永远排在最前）。全局可用 block 数 B 不足时求解：

```
max  Σ_r V_r(b_r)
s.t. Σ_r b_r ≤ B,   b_r ≥ pinned_r,   b_r ≥ floor_r
```

**为什么贪心即最优**：`V_r` 是**凹函数**（降序前缀和的边际增量单调不增），所以经典 water-filling 成立——每次把一个 block 配额给当前**边际价值最高**的请求；逆操作（回收）则每次从**边际价值最低**的请求收走一个 block。堆上做 lazy 贪心，复杂度近线性。

**两条由 OS 类比直接导出的硬约束**（答辩亮点：说明类比是认真的）：
1. **优先级必须带 aging**：纯语义优先级会饿死低密度租户——这正是 OS 调度的经典 starvation 教训。把它写进设计反而强化了类比的严肃性。
2. **tenant 是预算主体，request 是回收执行单元**：租户优先级定 cgroup 层 budget（慢时钟），block 语义分数定租户内回收顺序（页面回收层，快时钟）。

## 5. 与已有工作的差异化（Positioning）

| 已有工作 | 它占的坑 | 与 SemServe 的边界 |
|---|---|---|
| **KVShare** (2503.16525) | 跨请求语义**复用/去重** | 我们做**分配**不做复用；复用是可选扩展（IPC 类比） |
| **Semantic Scheduling** (2506.12204) | **请求级**语义排序 | 我们做 **token/block 级**跨请求分配，粒度更细 |
| **SAECache** (2605.18825) | **prefix cache** 的语义淘汰 | 我们动的是**显存竞争下的活跃请求 KV**，不是 prefix 缓存 |
| **RoleKV** | **角色级**淘汰 | 我们是连续语义分数 + 跨请求全局分配 |
| **Continuum/CacheTTL** (2511.02230) | **TTL** 保留 | 我们用语义价值替代时间/recency 作回收信号 |
| **SnapKV / StreamingLLM / H2O** | **单请求**无差别压缩 | 我们的核心是**跨请求**差异化分配（baseline 之一） |

> **一句话差异化**：唯一在"显存竞争 + 多租户 + token/block 级 + 语义价值驱动 + 渐进压缩替代整请求抢占"这五个维度同时成立的系统。任何阶段若发现实现滑向"语义复用"或"请求级排序"，立即停下来重新对齐。

## 6. 实验设计与 Baseline ⭐（答辩重点）

实验分三档：**保底（答辩必出）→ bonus（答辩加分）→ 完整矩阵（论文）**。

### 6.1 `[已可跑/保底]` 答辩保底实验：block 粒度质量 gate
**目的**：证明"把 token 级语义淘汰粗化到 paged-KV 的 block 粒度后，质量损失可接受"——这是整个 vLLM 路线的算法前提（Phase 1 go/no-go gate）。跑的是已在前序工作验证过的现有代码，换机器即可，**几乎不可能产不出结果**。

```bash
# token 级 vs block-16 vs block-32，固定 25% 预算
python eval_niah.py --policy semantic       --budget 0.25 --output results/v3/niah_token.json
python eval_niah.py --policy block_semantic --eviction-block-size 16 --budget 0.25 --output results/v3/niah_block16.json
python eval_niah.py --policy block_semantic --eviction-block-size 32 --budget 0.25 --output results/v3/niah_block32.json
```
**产出页**：质量-粒度曲线（x = block size，y = NIAH 准确率）。**判定**：block-16 ≥ token 级质量的 95% → 路线成立。
**可选加强**：budget sweep（0.1/0.3/0.5/1.0）、multi-needle、不同 needle position 的热力图。

### 6.2 `[bonus/高价值]` vanilla vLLM 抢占病理图
**目的**：用真实数据证明"vLLM 整请求抢占在压力下确实痛"——最强的动机页。
```bash
# 在 4×6000D 上装好 vLLM，跑最小压测
python bench/baseline_smoke.py --concurrency 4   --prompt-tokens 8000
python bench/baseline_smoke.py --concurrency 16  --prompt-tokens 8000
python bench/baseline_smoke.py --concurrency 64  --prompt-tokens 8000   # grep 日志里的 "preempt"
```
**产出页**：arrival rate / 并发 ↑ → 抢占次数 ↑、TTFT p99 ↑ 的拐点曲线。
**风险**：Blackwell（sm_120）需 CUDA 12.8+ / torch cu130，4 天内是主要不确定性。**拿不到不阻塞答辩**——退回用 vLLM 已知抢占行为 + PagedAttention 论文论证动机。

### 6.3 `[未来实验/论文]` 核心对照实验（the money shot）
**目的**：证明 SemServe 的语义渐进压缩**优于**整请求抢占。
```
工况：§6.2 找到的抢占触发负载（长 prompt 高并发）
对照：vanilla vLLM（整请求抢占）  vs  SemServe（语义渐进压缩）
预期：SemServe 的 TTFT p99 大幅下降（无重算），吞吐持平或更高，
      质量探针请求的正确率下降可控（< 5 pt）
判定（Phase 3 出口）：TTFT p99 改善 > 30% 且质量损失可控
```

### 6.4 Baseline 清单 `[未来实验/论文]`
全部以"vLLM 启动参数 / 小 patch"形式实现，跑**同一条 trace**：

| Baseline | 实现方式 | 想证明的对比点 |
|---|---|---|
| **vanilla vLLM** | 默认 FCFS + 抢占重算 | SemServe vs 现状 |
| **uniform 压缩** | 每请求无差别 budget=0.5（SnapKV/StreamingLLM 式，per-request） | "语义**跨请求**分配" 优于 "无差别压缩" |

| **oracle 上界** | 离线 drop/restore oracle 标注的 block 重要性替代 policy 分数 | 与"完美语义信号"的差距（policy 质量上界） |

### 6.5 评估指标 `[未来实验/论文]`
- **TTFT** p50/p99、**TPOT**、吞吐、抢占次数；
- ⭐ **quality-aware goodput**（本工作的新指标）= 比例(SLO 达标 ∧ 质量探针答对)——把"快"和"对"统一进一个指标；

## 7. 风险表与备选路线（答辩 Q&A 必问，提前备好）

| 风险 | 触发点 | 备选路线 |
|---|---|---|
| block 粒度质量崩 | §6.1 gate 不过 | block-8（vLLM 支持 block_size=8）；或路线 B |
| **block-table 压缩在所有 backend 都不可行** | Task 2.1 spike | **路线 B**：放弃 token/block 级淘汰，改做"**请求级语义压缩档位**"——压力下把低密度请求整体 swap 到 CPU int8（粗粒度但工程量小一个量级，故事改讲 semantic-aware preemption/swap ordering） |
| attention proxy 拿不到（vLLM 无 `output_attentions`） | Task 2.2 spike | SnapKV 式 prefill 后 query-window 打分；再不行退回纯静态信号 + OP MLP |
| money shot 信号弱 | Phase 3 出口 | 加大 contention / 提高长文租户占比；或换工况（agent 多轮、超长 RAG） |

> **关键诚实点**：`attention_tracker.py` 依赖 `output_attentions`，**不可移植到 vLLM**——这是已知的硬约束，替代方案（SnapKV 式 query-window 打分 / 纯静态信号）已经在计划里。主动讲出来比被问出来好。

## 9. 诚实边界：已完成 vs 设计提案

- **`[已验证]` 现在就有的**：semanticache/OP-SieveKV 的语义淘汰在 HF testbed 上有效（前序工作）；`semserve.block_mapping`（segment→block 映射 + 整块 keep 选择）已实现并通过单测；`block_semantic` 淘汰策略已接入 `eval_niah.py`，可直接产出 §6.1 的质量曲线。
